# 실시간 백그라운드 스캔 기반 회피/탈출 알고리즘 — Design

## 배경 / 문제

`AUTONOMOUS_FORWARD_LIDAR_EXAMPLE.md`의 현재 회피/탈출 알고리즘은 "멈춰서 64칸을 다
읍고(약 0.6~1초) → 판단 → 회전/전진 명령(블로킹, 완료까지 1~2초) → 다시 멈춰서 읍기"를
반복하는 **정지-스캔-판단-이동**의 순차 구조다. 실측 결과(라디오 디버그 로그) 이 구조
자체는 기준값(배경) 비교 도입 이후 크게 개선됐지만, 다음 한계가 남아 있다:

- 회전/전진 중에는 센서를 다시 볼 수 없어, 회전하는 동안 갑자기 가까워지는 장애물을
  즉시 못 잡는다.
- 회전 탐색이 45°/15° 같은 고정 각도 목록을 순회하는 구조라, "이미 충분히 트인 곳을
  지나쳐서 더 멀리까지 다 돈 뒤에야" 멈춘다.
- 매번 64칸을 동기적으로 다 읍어야 다음 판단을 할 수 있어, 판단 주기 자체가 느리다.
- 라이다만으로는 못 보는 장애물(유리 등)에 막혔을 때 이를 구분할 방법이 없었다.

이 설계는 micro:bit의 협동형(cooperative) 백그라운드 실행을 이용해 센서 읍기와 판단을
분리하고, 회전을 끊어서 멈추는 대신 PID 회전을 "끼어들 수 있게"(non-blocking) 돌리면서
매 틱마다 실시간으로 확인하다가 충분한 공간이 보이면 그 자리에서 즉시 멈추는 방식으로
바꾼다.

## 범위

- `AUTONOMOUS_FORWARD_LIDAR_EXAMPLE.md`의 회피/탈출 알고리즘(데이터 레이어 포함)을
  이 설계로 **대체**한다. 새 파일을 만들지 않고 같은 파일을 크게 고친다(LCD/로그/버튼/
  초기화 등 기존 골격은 재사용).
- 백그라운드 fiber로 64칸 라이다 캐시를 계속 갱신
- 기준값(배경) 비교는 그대로 유지하되, 정지-스캔이 아니라 캐시에 적용
- 직전 틱 대비 변화량(delta) 기반 판단을 평상시 회피와 "헛돌이/안 보이는 장애물에
  막힘" 감지에 사용
- `pidControlAngle(.., Allowed)` + `pidControlStop()`을 이용한 끼어들 수 있는 연속
  회전 탐색(랜덤 좌/우 시작 방향)
- 5단계 우선순위 결정 루프
- 점진적 적응형 정밀도(연속 성공 기준, 빠른 모드에서도 주기적으로 맨 아래 행 포함)
- 디버그 콘솔 강화(레벨 조절 가능)

## 범위 외

- 모터 전류/과부하 직접 측정 — `maqueenPlusV3.ts`에 해당 API가 없어 불가능. 대신
  라이다 delta로 "헛돌이/막힘"을 간접 판단한다(아래 참조).
- 좌우 바퀴 엔코더 속도 비교(`readRealTimeSpeed`)를 통한 비대칭 슬립 감지 — 라이다
  delta 기반 판단으로 통일하면서 범위 밖으로 뺀다(설계 단순성 우선, 필요해지면 이후
  별도 작업으로 추가 가능).
- 백그라운드 캐시의 완벽한 시간 동시성 보장 — 64칸이 한 번에 찍힌 스냅샷이 아니라
  칸마다 갱신 시점이 다른 점을 감수한다(아래 트레이드오프 참조).
- 1~5순위를 모아 랜덤 가중 선택하는 방식 — 연속 회전 탐색 구조에서는 "처음 찾는 즉시
  멈춤"이 기본이라 후보를 모을 수 없으므로, 회전 시작 방향(좌/우)만 랜덤화한다.

## 핵심 트레이드오프

백그라운드 fiber가 64칸을 한 바퀴 도는 데도 여전히 시간이 걸린다(메인 루프와 협동
스케줄링으로 시간을 나눠 쓰기 때문에 더 걸릴 수 있음). 그래서 메인 루프가 매 틱
읍는 캐시는 "완벽하게 동시에 찍힌 스냅샷"이 아니라 **칸마다 최신 갱신 시점이 다른
값들의 모음**이다. 정지-스캔 방식보다 실시간성은 훨씬 좋아지지만, 완벽한 동시성은
포기한다. 디버그 콘솔에 캐시 신선도(`cacheAge`)를 노출해 이 트레이드오프가 실제로
문제가 되는지 운영 중에 확인할 수 있게 한다.

## 아키텍처

```
[백그라운드 fiber]                      [메인 포그라운드 루프, 매 틱 40ms]
matrixPointOutput()                     1. 캐시 + 기준값 → 셀판정(배경/장애물)
  64칸 순회                              2. 직전 판정과 비교 → delta 계산
  → 캐시[64] 갱신                        3. 우선순위 1~5 평가, 동작 실행
  (셀마다 갱신시각도 기록)                 4. 적응형 정밀도로 "이번 틱 검사 범위" 결정
  매 셀 basic.pause(10)로 양보            5. 디버그 로그 출력(레벨별)
```

두 fiber는 micro:bit의 협동형 스케줄러 위에서 동작한다(`basic.forever()`도 내부적으로
`control.inBackground()`로 구현됨). `matrixPointOutput()`이 셀마다 `basic.pause(10)`을
호출하기 때문에 그 시점마다 다른 fiber로 제어권이 넘어가, 백그라운드 스캐너와 메인
루프가 서로 끼어들며 진행된다.

## 컴포넌트 설계

### 1. 백그라운드 스캐너

```typescript
let 캐시: number[] = []          // 64칸 raw 값 (row*8+col)
let 캐시갱신시각: number[] = []   // 칸별 마지막 갱신 시각(ms)

control.inBackground(function () {
    while (true) {
        for (let row = 0; row < 8; row++) {
            for (let col = 0; col < 8; col++) {
                캐시[row * 8 + col] = matrixLidarDistance.matrixPointOutput(라이다주소, col, row)
                캐시갱신시각[row * 8 + col] = input.runningTime()
            }
        }
    }
})
```

루프 안에 별도 `basic.pause()`를 추가하지 않는다 — `matrixPointOutput()` 자체가 매
호출마다 내부적으로 양보하므로 충분하다.

### 2. 기준값(배경) 분류 — 기존 로직 재사용

`기준값측정()`/`셀판정()`은 기존과 동일한 의미를 유지하되, 이제 **캐시를 입력으로**
받는다. 즉시 라이다를 다시 읍지 않고 캐시에서 읍는다는 점만 다르다. 8행 전체를 매번
판단에 쓸 수 있다(바닥 행도 기준값과 비교하므로 안전).

### 3. Delta 추적기

```typescript
let 직전판정: number[] = []   // 셀판정() 결과의 이전 틱 스냅샷(64칸)

function 델타확인(칸인덱스: number, 지금판정: number): number {
    let 이전 = 직전판정[칸인덱스]
    if (이전 == undefined || 이전 < 0 || 지금판정 < 0) return 0  // 모름은 비교 불가
    return 이전 - 지금판정   // 양수 = 가까워짐
}
```

두 가지 용도로 쓴다:
- **긴급 위험**: 특정 칸(주로 정면 중앙)의 delta가 한 틱 사이 `긴급delta한계mm`
  이상이면(절대 거리와 무관하게) 즉시 정지·후진 우선순위로 분류.
- **헛돌이/안 보이는 장애물 감지**: 전진 명령을 내려 진행 중인데, 기대되는 만큼
  거리가 안 줄어들면(여러 틱에 걸친 delta 합이 `최소진행mm` 미달) "라이다로 안 보이는
  것에 막혔거나 헛돌고 있다"로 판단하고 다른 방향을 시도한다. 모터 엔코더
  (`readRealTimeSpeed`) 비교는 쓰지 않는다(범위 외).

### 4. 연속 회전 탐색

```typescript
function 회전탐색시작(): void {
    let 방향 = Math.random() < 0.5 ? 1 : -1   // 좌/우 랜덤
    회전탐색방향 = 방향
    maqueenPlusV2.pidControlAngle(방향 * 170, maqueenPlusV2.MyInterruption.Allowed)
}
```

메인 루프는 회전 탐색 상태일 때 매 틱:
1. 캐시 기반으로 "지금 향한 방향에 충분한 공간이 있는가"(정면 중앙열 기준, 기존
   `정면안전()`류 판단 재사용) 확인.
2. 있으면 `maqueenPlusV2.pidControlStop()`으로 즉시 정지 → 평상시 전진으로 전환.
3. 없고, `pidControlAngle()` 1회 호출(최대 170도)이 다 돌아 끝났을 것으로 추정되는
   시간이 지났으면 같은 방향으로 한 번 더 호출(누적 회전각을 시간 기반으로 추정해
   `360도 다 돌았는지`를 판단 — 5단계 최종 폴백 전환 기준).
4. 4단계(일정 시간/범위 이상 못 찾음) → 5단계(360도 전체에서 가장 나은 곳, 회전하며
   계속 점수 추적 후 최고점 방향으로 정렬)로 전환. 구체적으로는 `pidControlAngle(방향
   * 170, Allowed)` 호출을 같은 방향으로 최대 3회까지 이어서 호출(약 510도, 360도를
   넉넉히 커버)해도 "충분한 공간"을 못 찾으면 4→5단계로 넘어간다.

### 5. 우선순위 결정 루프 (매 틱)

1. **긴급 정지/후진** — delta 기반 급격한 근접 감지, 모든 동작 중단 후 최우선 처리
2. **회전 탐색 중 충분한 공간 발견** — 즉시 정지 후 전진 전환
3. **평상시 전진 중 정면 막힘 감지** — 정지 후 회전 탐색(랜덤 방향) 시작
4. **회전 탐색이 일정 시간/범위 이상 못 찾음** — 헛돌이/막힘 신호와 결합해 전략 전환
5. **그래도 답이 없음** — 360도 전체를 훑어 가장 나은 방향으로 정렬 후 전진

### 6. 적응형 정밀도

```typescript
let 정밀도레벨 = 0          // 0(천천히/꼼꼼) ~ 최대레벨(빠름/간단)
let 연속성공틱 = 0
```

- 회피 없이 전진이 연속 `정밀도증가조건`회 성공하면 `정밀도레벨` 1단계 증가(점진적,
  기존 `적응전진거리cm` 패턴과 동일한 점진 증가/감쇠 스타일).
- 단순 회피 성공 시 정밀도레벨을 소폭 감소, 회전 탐색(탈출) 트리거 시 최저 레벨로
  완전 리셋 — 기존 "실패 시 조금 깎기 vs 5연속 실패 시 탈출"의 2단계 감쇠 구조를
  그대로 따른다.
- 레벨에 따라: 낮을수록(천천히/꼼꼼) 8행 전체 + 더 자주 delta 확인, 높을수록(빠름)
  중앙 위주로 가볍게 보되, **빠른 모드에서도 N틱마다 한 번은 맨 아래 행을 포함**해
  근접한 낮은 장애물을 놓치지 않는다.
- 레벨 곡선 자체에는 랜덤성을 넣지 않는다(안전 관련 페이싱은 예측 가능하게 유지).
  랜덤성은 회전 탐색 시작 방향에만 적용한다.

### 7. 디버그 콘솔 강화

- `디버그레벨` 상수(0=끄기, 1=하트비트만, 2=하트비트+델타/타이밍 전부)로 무선 송신
  비용을 조절한다.
- 하트비트에 `priority`(발동한 우선순위 단계), `precision`(정밀도레벨),
  `cacheAge`(캐시 중 가장 오래된 칸의 갱신 경과 ms) 추가.
- 백그라운드 스캐너가 64칸 한 바퀴 도는 데 걸린 시간(ms)을 매 바퀴 끝에 로그.
- 델타 트리거 발생 시 어떤 칸에서 얼마나 변했는지 로그.
- 회전 탐색 시작 시 선택된 방향, 멈춘 시점까지 걸린 시간, 멈춘 시점 점수 로그.

## 에러 처리 / 안전 기본값

- 캐시에 아직 값이 없는 칸(부팅 직후 등)은 `-1`(모름)로 취급, 보수적으로 막힘 처리.
- `셀판정()`의 기존 글리치(raw==0)/센티널(raw>=4000) 구분 규칙을 그대로 유지한다.
- delta 비교는 두 값 중 하나라도 "모름(-1)"이면 비교하지 않고 0(변화 없음)으로 처리.
- 5단계(360도 전체 탐색)에서도 진짜 답이 없으면(최고점이 0 이하) 기존과 동일하게
  `"NO ESCAPE"` 상태로 정지하고 X 아이콘을 표시한다.

## 기본 조정값 (초기값, 실측 후 튜닝 가능)

| 변수 | 초기값 | 의미 |
|---|---:|---|
| `긴급delta한계mm` | 80 | 정면 칸이 한 틱 사이 이만큼 이상 가까워지면 절대 거리와 무관하게 긴급 정지·후진 |
| `진행확인시간ms` | 1000 | 이 시간 동안의 delta 합을 봐서 "헛돌이/막힘" 여부를 판단 |
| `최소진행mm` | 20 | 전진 명령 중 `진행확인시간ms` 동안 최소 이만큼은 가까워져야(또는 멀어져야) "정상 진행"으로 인정 — 미달이면 헛돌이로 판단 |
| `정밀도레벨개수` | 3 | 0(천천히/꼼꼼)~2(빠름/간단) 3단계 |
| `정밀도증가조건` | 3 | 회피 없이 연속 이 횟수만큼 전진 성공하면 정밀도레벨 1단계 증가(기존 `전진성공증가조건`과 동일 스타일) |
| `빠른모드맨아래행주기틱` | 5 | 정밀도레벨이 최고일 때도 5틱마다 한 번은 맨 아래 행(row7)을 포함해 확인 |
| `회전1회각도` | 170 | `pidControlAngle()` 1회 호출당 회전각(최대 180 제한 내) |
| `회전탐색최대반복` | 3 | 같은 방향으로 `회전1회각도`를 최대 이 횟수까지 이어 호출(약 510도)해도 못 찾으면 4→5단계 전환 |

## 검증

자동 테스트 러너가 없는 프로젝트라 기존과 같은 패턴을 따른다.

1. 정적: `tsc --noEmit` (코드 블록 추출 + tsconfig 임시 수정 절차, 기존과 동일)
2. 의사실행 트레이스: 우선순위 1~5단계가 각각 올바른 입력 조건에서 발동하는지 표로
   확인. delta 계산식이 "모름" 케이스를 안전하게 처리하는지 확인.
3. 하드웨어 체크리스트(사람이 실행):
