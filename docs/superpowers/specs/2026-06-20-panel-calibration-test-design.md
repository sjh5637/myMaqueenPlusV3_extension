# 차폭 패널 기준 거리/각도 보정-테스트 스크립트 설계

## 배경 / 목적

`AUTONOMOUS_WANDER_EXAMPLE.md`의 메인 자율주행 코드는 빈 공간을 기준으로
하는 `바닥보정()`만 가지고 있다. 실제 거리 정답(ground truth)이 없어서
"라이다가 실제로 몇 mm를 몇 mm로 읽는지", "센서가 인식 가능한 최소 거리는
몇 mm인지", "값이 비정상적으로 튀는 지점이 있는지", 그리고 "좌/중앙/우
빔이 차체 폭(15cm) 전체에서 일관되게 정확한지"를 정량적으로 알 수 없다.

5cm 정육면체 나무 블록 9개를 조합해 **가로 15cm × 높이 15cm × 두께 5cm
패널을 세워서** 바닥에 고정해두면, 로봇 차체 폭(15cm)과 패널 폭이 같아져
좌측/정면/우측 빔이 동시에 같은 평면을 보게 된다. 패널을 세워서 높이를
15cm로 만드는 이유는 라이다 장착 높이 범위(100~180mm) 전 구간에서 패널이
계속 시야에 걸리도록 하기 위함이다(5cm 높이로 눕혀두면 거리가 멀어질수록
빔이 패널 위를 넘어가 바닥을 볼 수 있음).

로봇이 패널에 코를 맞댄 상태(거리 0)에서 시작해 1cm씩 정확히 후진하며
라이다(매트릭스 8x8)와 전면 초음파(P13 TRIG / P14 ECHO)를 동시에
측정하면, 알려진 실제 거리와 센서 값을 좌/중앙/우 각 구역별로 직접
비교할 수 있다. 이 설계는 메인 주행 코드와는 완전히 분리된, 1회성 벤치
테스트용 스크립트를 다룬다.

## 비목표 (Out of scope)

- 메인 자율주행 코드(`AUTONOMOUS_WANDER_EXAMPLE.md`)는 수정하지 않는다.
- 라이다 높이(A+B) 자동 스윕은 하지 않는다 — 필요하면 높이만 바꿔 같은
  테스트를 재실행한다.
- LCD 레이더맵 등 무거운 화면 갱신은 넣지 않는다 — 측정 정밀도/속도가
  우선이며, 5x5 LED 매트릭스로 진행 상황 숫자만 표시한다.

## 하드웨어 전제

- 5cm 정육면체 9개를 3x3로 쌓아 가로 15cm × 높이 15cm × 두께 5cm 패널을
  만들고, 로봇 정면에 **세워서** 바닥에 고정(로봇에는 붙이지 않음).
- 패널 폭(15cm)이 로봇 차체 폭과 같으므로, 로봇을 패널 정면 중앙에
  가능한 한 똑바로 맞대어 거리 0 위치에 둔다.
- 매트릭스 라이다: `matrixLidarDistance.Addr.Addr4` (0x33), 8x8 그리드.
- 전면 초음파: TRIG = `DigitalPin.P13`, ECHO = `DigitalPin.P14`
  (`maqueenPlusV2.readUltrasonic(P13, P14)`, 단위 cm).
- 라디오: 그룹 77 (메인 코드의 디버그 콘솔과 동일 채널/리시버 재사용 가능).

## 동작 흐름

1. 부팅 시 `로봇초기화_테스트()`: I2C init, 라이다 init, 라디오 그룹 77,
   송신 출력 설정. 5x5 매트릭스에 대기 아이콘 표시.
2. B 버튼을 누르면 시작.
3. 3, 2, 1 카운트다운 (5x5 매트릭스 숫자 표시).
4. 스텝 0 ~ 30 (cm, 총 31스텝)을 순회:
   - 스텝 0: 후진 없이 그 자리에서 측정 (패널에 거의 붙어있는 상태 —
     센서 최소 인식 거리 추정용. 0/무효값이 나올 수 있음을 정상으로
     간주).
   - 스텝 1~30: `maqueenPlusV2.pidControlDistance(SpeedCCW, 1,
     MyInterruption.NotAllowed)`로 정확히 1cm 후진 후 정지, 측정.
5. 30스텝 완료 후 `테스트요약()` 실행, 라디오로 요약 전송, 5x5 매트릭스에
   완료 아이콘 표시.

## 스텝당 측정 로직

함수 `스텝측정(스텝번호: number)`:

1. 8x8 그리드를 **3회 반복 읍기**(노이즈 확인용). 각 셀(x,y)마다 3개
   샘플의 **중앙값(median)** 을 최종 그리드 값으로 채택.
2. 패널이 차폭 전체를 덮으므로, 메인 코드의 좌/정면/우 구역 컬럼과
   동일한 위치에서 3개 대표값을 뽑는다:
   - `좌값` = x=1 컬럼, y=3..4 행의 median (메인 코드 좌측 빔과 동일
     위치)
   - `중앙값` = x=3..4 컬럼, y=3..4 행의 median (메인 코드 정면 빔과
     동일 위치)
   - `우값` = x=6 컬럼, y=3..4 행의 median (메인 코드 우측 빔과 동일
     위치)
   - 각 값에 대해 3회 샘플 중 **최대-최소 범위(노이즈)** 도 계산 →
     `노이즈좌`, `노이즈중`, `노이즈우`.
3. `maqueenPlusV2.readUltrasonic(P13, P14)`로 초음파 1회 읽기(cm) →
   mm로 환산 → `초음파값` (초음파는 정면 1방향만 측정하므로 좌/우와는
   비교하지 않고 `중앙값`과만 비교).
4. `기대거리mm = 스텝번호 * 10`.
5. 오차 계산: `d좌 = 좌값 - 기대거리mm`, `d중 = 중앙값 - 기대거리mm`,
   `d우 = 우값 - 기대거리mm`, `dUson = 초음파값mm - 기대거리mm` (0/무효
   측정이면 오차는 표시하지 않고 `N/A`로 표기).
6. 점프 판정: 좌/중/우 각각 직전 스텝 값(`이전좌값`/`이전중앙값`/
   `이전우값`, 모듈 전역 변수)과 비교. `|현재값 - 이전값| >
   큐브테스트_점프임계mm`(기본 25mm, 기대 변화량 10mm의 2.5배)이면 해당
   구역에 `JUMP`로 표시하고 점프 스텝 목록에 구역과 함께 기록.
7. 라디오 전송 (총 9줄):
   - 헤더: `STEP {s} cm={s} exp={기대거리mm}mm L={좌값}mm
     C={중앙값}mm R={우값}mm uson={초음파값mm}mm dL={d좌} dC={d중}
     dR={d우} dU={dUson} nL={노이즈좌} nC={노이즈중} nR={노이즈우}
     {JUMP:구역 또는 빈 문자열}`
   - 그리드 8줄: `ROW{y}: {v0} {v1} ... {v7}` (median 그리드 값, mm,
     0 = 미인식)
8. `이전좌값`/`이전중앙값`/`이전우값`을 현재 값으로 갱신.
9. 다음 비교를 위해 스텝 결과(`스텝번호`, `좌값`, `중앙값`, `우값`,
   `d좌`, `d중`, `d우`, `dUson`, 구역별 `JUMP 여부`)를 배열에 누적
   저장.

## 종료 후 요약 (`테스트요약()`)

축적된 31개 스텝 결과를 좌/중앙/우 구역별로 각각 바탕으로:

- **구역별 최소 인식 거리 추정**: 해당 구역 값이 처음으로 0이 아니고,
  다음 스텝에서도 0이 아니며 두 값의 차이가 `큐브테스트_점프임계mm`
  이내로 안정된 첫 스텝의 `기대거리mm`.
- **구역별 최대 오차 스텝**: `|d좌|`/`|d중|`/`|d우|`가 각각 가장 큰
  스텝과 그 값.
- **좌우 비대칭 확인**: 같은 스텝에서 `d좌`와 `d우`의 차이가 크면(패널
  폭과 차체 폭이 같으므로 이상적으로는 비슷해야 함) 라이다 장착이
  좌우로 비뚤어졌을 가능성을 의미 — 가장 차이가 큰 스텝을 보고.
- **점프 스텝 목록**: 구역별로 `JUMP`로 기록된 모든 스텝 번호.

이 항목들을 라디오로 구역별 한 줄씩 전송하고, 5x5 매트릭스에
`basic.showIcon(IconNames.Yes)`를 표시해 테스트 완료를 알린다.

## 에러 처리

- 라이다/초음파가 0 또는 비정상 값을 반환해도 스크립트는 멈추지 않고
  해당 스텝을 `dLidar`/`dUson` = `N/A`로 기록하며 계속 진행한다(센서
  한계를 찾는 것이 테스트의 목적이므로 무효값 자체가 유효한 데이터).
- `pidControlDistance` 후진이 예상보다 일찍 멈추는 경우(엔코더 슬립
  등)는 이번 설계에서는 별도 보정을 하지 않는다 — 이 테스트는 정밀
  주행 검증이 아니라 센서 응답 특성 파악이 목적이며, 실제 이동거리와
  명령 거리의 오차는 추후 별도로 다룬다.

## 테스트 / 검증 방법

- 패널 위치를 줄자/마퀸 휠 회전수 등으로 0, 10, 20, 30cm 지점에 표시해
  두고, 스크립트 실행 중 줄자로 로봇의 실제 위치가 예상 스텝과
  일치하는지 육안으로 대조한다.
- 별도 수신기 마이크로비트(기존 라디오 디버그 콘솔, 그룹 77)로 전체
  31스텝 × 9줄 로그를 캡처해, 사용자가 그대로 복사해 전달하면 분석할
  수 있다.
- 1회 실행 결과를 통해: (1) 구역별 센서 최소 인식 거리, (2) 정상 작동
  구간의 좌/중앙/우 평균 오차, (3) 좌우 비대칭(마운트 틀어짐) 여부,
  (4) 튀는 지점의 패턴(특정 거리 구간/구역에서만 발생하는지)을 확인하는
  것이 성공 기준이다.
