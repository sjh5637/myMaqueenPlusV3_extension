# LCD 8x8 레이더맵을 5x5 LED 표시로 교체 — Design

## 배경 / 문제

`AUTONOMOUS_WANDER_EXAMPLE.md`의 `lcd레이더맵표시()`는 DFR0997 LCD에 8x8(64칸) 사각형을
I2C로 그려서 라이다 전체 빔을 시각화한다. 칸마다 I2C 청크 전송 + `basic.pause(5ms)`가
들어가 있어 셀이 많이 바뀌는 틱에서는 그리는 데 걸리는 시간이 누적되어 메인 루프를
지연시킨다(`lcd레이더맵표시()` 안의 `MAP drew.. ms..` 디버그 로그가 이미 이 비용을
측정해서 보여주고 있었음). 이 기능을 통째로 제거하고, 같은 정보(좌/정면/우 장애물
근접도)를 micro:bit 본체의 5x5 LED로 대체한다. LCD의 텍스트 상태 표시(`lcd문자(...)`
5줄)는 그대로 유지한다.

## 범위

- 제거: 8x8 레이더맵 관련 코드 전부(LCD 텍스트는 그대로 둠)
- 추가: 5x5 LED 중앙 한 줄(row 2)에 좌/정면/우 밝기 표시
- 갱신: 메인 루프 매 틱(약 40ms)마다 갱신, 추가 센서 읽기/I2C 없음(이미 읽은 값 재사용)
- 문서: `AUTONOMOUS_WANDER_EXAMPLE.md`의 설명/표/체크리스트를 새 동작에 맞게 갱신

## 제거 대상

| 종류 | 이름 |
|---|---|
| 함수 | `lcd레이더맵표시()`, `lcd사각형()`, `거리색()` |
| 전역 변수 | `LCD맵사용`, `LCD맵이전색`, `마지막맵시각` |
| 전역 상수 | `LCD맵갱신간격ms`, `LCD맵X`, `LCD맵Y`, `LCD맵칸`, `LCD맵간격`, `LCD맵ID시작` |
| 호출 | `lcd표시()` 안의 `lcd레이더맵표시(false)` |

`지점읽기()`, `lcd명령쓰기()`, `LCD맵칸쓰기지연ms`는 다른 곳(기울기보정, 일반 LCD
텍스트 출력)에서도 쓰이므로 그대로 둔다.

## 추가 설계

### `거리밝기(거리mm: number): number`

기존 `거리색()`과 동일한 거리 구간을 그대로 재사용해 밝기(0~255)로 변환한다.

```typescript
function 거리밝기(거리: number): number {
    if (거리 == 0) return 0
    if (거리 < 긴급정지거리mm) return 255
    if (거리 < 320) return 200
    if (거리 < 520) return 130
    if (거리 < 800) return 70
    return 20
}
```

(0=무효/꺼짐, 가까울수록 밝게. `거리색()`의 5단계 색상 구간과 1:1 대응.)

### `LED레이더표시(): void`

```typescript
function LED레이더표시(): void {
    basic.clearScreen()
    led.plotBrightness(0, 2, 거리밝기(구역최소(0)))
    led.plotBrightness(1, 2, 거리밝기(구역최소(0)))
    led.plotBrightness(2, 2, 거리밝기(정면최소거리()))
    led.plotBrightness(3, 2, 거리밝기(구역최소(2)))
    led.plotBrightness(4, 2, 거리밝기(구역최소(2)))
}
```

- 열 매핑: 0,1열=좌(`구역최소(0)`), 2열=정면(`정면최소거리()`, 초음파 융합값 — 비상정지
  판단과 동일한 값을 보여줘서 LED와 실제 정지 사유가 일치함), 3,4열=우(`구역최소(2)`)
- row 0,1,3,4는 항상 꺼짐(매 호출마다 `clearScreen()` 후 row 2만 켬) — 보정 성공/실패
  아이콘(`basic.showIcon`)이나 카운트다운(`basic.showNumber`)이 같은 화면을 잠깐
  덮어써도, 다음 `LED레이더표시()` 호출이 곧바로 깨끗하게 다시 그림
- 인자로 받는 거리는 이미 그 틱에 읽어둔 `최근값`/`최근초음파mm` 기반 계산값이라 추가
  I2C 호출이 없음(원가 거의 0)

### 호출 시점

메인 루프(`basic.forever`)에서 `감시샘플읽기()` → `막힘연속갱신()` 다음 줄에
`LED레이더표시()`를 추가한다. 이 둘은 주행 중(`주행시작됨 == true`)에만 매 틱(약
40ms = `루프대기ms`) 실행되는 구간이므로, LED도 같은 주기로 거의 실시간 갱신된다.
기존 8x8 맵의 1000ms 스로틀(I2C 비용 때문에 필요했던 제약)이 필요 없어진다.

### 대기 화면 정리

`lcd대기표시()`의 스로틀 통과 분기(이미 있는 "너무 자주 호출되면 return" 체크 다음)에
`basic.clearScreen()` 한 줄을 추가한다. 주행이 멈추고 대기 화면으로 돌아오면(높이 변경,
BASE NOISY 정지, 출발/탈출 탐색 실패 등 `주행시작됨 = false`가 되는 모든 경로) 다음
대기 화면 갱신 시점(최대 `LCD갱신간격ms`=500ms 이내)에 LED에 남아있던 레이더 막대가
자동으로 꺼진다. 별도의 정지 시점 후킹 없이 기존 대기 화면 갱신 경로에 얹는 것이라
변경 폭이 작다.

## 문서 갱신

`AUTONOMOUS_WANDER_EXAMPLE.md`에서 다음을 갱신한다.

- 상단 "현재 권장 사용 흐름" 설명 중 "LCD 레이더맵은 저빈도 갱신으로 8x8 전체를
  시각화" 문구 → 5x5 LED 표시로 교체되었음을 설명
- 조정값 표에서 `LCD맵사용`/`LCD맵갱신간격ms` 행 제거
- "핵심 변화" 11번 항목(LCD 레이더맵 갱신 속도 개선) → 5x5 LED 교체로 재작성하거나 제거
- 테스트 체크리스트 6번("LCD 왼쪽 8x8 색상 맵이 느리면 끄고 테스트") → 5x5 LED 확인
  항목으로 교체
- "## LCD 레이더맵 색상" 섹션 → "## 5x5 LED 밝기" 섹션으로 교체(밝기 단계 설명)

## 검증

이 프로젝트는 자동 테스트 러너가 없다(MakeCode/PXT 하드웨어 펌웨어). 검증은:

1. 정적: 코드 블록 추출 후 `tsc --noEmit`으로 타입 체크(`led.plotBrightness`,
   `basic.clearScreen` 등 core API 인식 확인)
2. 의사실행 트레이스: `거리밝기()` 경계값(0, 279, 280, 319, 320, 799, 800) 확인
3. 하드웨어 체크리스트(사람이 실행): 좌/정면/우 각각 장애물을 가까이 댔을 때 해당 LED
   칸이 밝아지는지, 장애물이 없으면 꺼지는지, 정지 후 대기 화면으로 돌아오면 LED가
   꺼지는지, 기존 8x8 맵 대비 메인 루프가 더 빠르게 도는지(체감)

## 범위 외

- LCD 텍스트 상태 표시(`lcd표시()`/`lcd대기표시()`의 `lcd문자(...)` 줄들)는 변경하지 않음
- 자율주행 의사결정 로직(전진/회피/탐색 방향 결정)은 변경하지 않음
- 5x5 LED의 다른 행(0,1,3,4)을 활용한 추가 정보 표시는 범위 외(향후 필요 시 별도 설계)
