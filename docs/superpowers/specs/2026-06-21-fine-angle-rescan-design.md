# 재탐색180() 다단계(굵게→세밀하게) 각도 탐색 — Design

## 배경 / 문제

`AUTONOMOUS_WANDER_EXAMPLE.md`의 `재탐색180(탈출모드: boolean)`(`:870`)은 정면 ±90°를
45° 간격 5방향만 보고 끝낸다. 여기서 안전한 방향을 못 찾으면(`방향안전확인()` 실패 또는
점수 미달) 곧바로 실패(`false`) 처리되어, 호출자(`출발준비()`, `탈출탐색()`)는 바로
`"NO SAFE DIR"`로 포기하고 `basic.showIcon(IconNames.No)`(X 표시)를 띄운다.

실제 하드웨어 테스트에서 라이다 각도 보정이 `ANGLE OK`가 아닌 상태로 출발을 시도하면
정면 부근이 전부 "막힘"으로 판정되어 45°/5방향으로는 안전한 틈을 못 찾고 곧장 실패하는
사례가 있었다. 더 세밀한 각도로 다시 보면 찾을 수 있는 좁은 틈을 놓치고 있을 가능성이
있다. 사용자 요청대로 "처음엔 굵게, 못 찾으면 세분화, 그래도 없으면 다음 90°씩 확장해서
360° 전체를 세밀하게, 그 후에야 실패 처리"하는 단계적 탐색으로 바꾼다.

## 범위

- `재탐색180()` 내부에 3단계를 추가해 총 4단계(기존 1 + 신규 3)로 확장
- 회전/점수 측정 루프를 `각도순회탐색()` 헬퍼로 일반화해 4단계 모두 재사용
- `출발준비()`, `탈출탐색()` 등 호출자 쪽 시그니처/호출 방식은 변경 없음
  (`재탐색180(탈출모드: boolean): boolean` 그대로)
- `초기360탐색()`(8방향 45° 풀서클 코스 스캔)은 변경하지 않음 — 1단계 코스 스캔은
  기존 그대로 두고, 정밀화는 `재탐색180()` 안에서만 일어남

## 단계 구성

| 단계 | 범위 | 간격 | 방향 수 | 각도 배열 |
|---|---|---|---|---|
| 1 (기존) | 정면 ±90° | 45° | 5 | `재탐색각도 = [-90,-45,0,45,90]` (기존, 변경 없음) |
| 2 (신규) | 정면 ±90° | 15° | 13 | `재탐색각도세밀 = [-90,-75,-60,-45,-30,-15,0,15,30,45,60,75,90]` |
| 3 (신규) | 우후방 105°~180° | 15° | 6 | `재탐색각도세밀후방우 = [105,120,135,150,165,180]` |
| 4 (신규) | 좌후방 -105°~-180° | 15° | 6 | `재탐색각도세밀후방좌 = [-105,-120,-135,-150,-165,-180]` |

단계 2~4를 합치면 정면 ±90°(단계2) + 우후방(단계3) + 좌후방(단계4) = 360° 전체를
15° 간격으로 덮는다. 각 단계가 끝나면 점수(`재탐색최소점수` 이상)와
`방향안전확인()`을 확인해 통과하면 그 방향을 그대로 채택하고 `true`를 반환한다.
실패하면 그 단계에서 돈 각도만큼 되돌려(`pidControlAngle(-최고각, ...)`) 원래 진입
방향으로 복귀한 뒤 다음 단계로 넘어간다. 4단계 모두 실패하면 `false`를 반환하며,
이는 호출자에서 기존과 동일하게 `"NO SAFE DIR"` / `"ESCAPE FAILED"` 처리(X 아이콘)로
이어진다 — 즉 "360° 전체를 세밀하게 다 봤는데도 안전한 방향이 없다"는 뜻일 때만
최종 실패가 뜬다.

## 설계: `각도순회탐색()` 공용 헬퍼

현재 `재탐색180()`은 회전 스텝이 `maqueenPlusV2.pidControlAngle(45, ...)`로 고정돼
있어 각도 배열 값과 실제 회전량이 분리돼 있다(배열 값은 라벨/최종 보정 계산용일 뿐).
이를 배열의 연속 항목 차이로 회전량을 계산하는 일반 함수로 추출한다.

```typescript
function 각도순회탐색(각도목록: number[]): number {
    let 최고점수 = -999999
    let 최고각 = 각도목록[0]
    maqueenPlusV2.pidControlAngle(각도목록[0], maqueenPlusV2.MyInterruption.NotAllowed)
    for (let i = 0; i < 각도목록.length; i++) {
        let 후보각 = 각도목록[i]
        let 점수 = 탐색점수(후보각)
        if (점수 > 최고점수) {
            최고점수 = 점수
            최고각 = 후보각
        }
        lcd문자(1, 8, 16, 감시모드 + " " + (i + 1) + "/" + 각도목록.length, 0x000000)
        lcd문자(2, 8, 54, "DIR " + 후보각, 0x0000ff)
        lcd문자(3, 8, 92, "SCORE " + 점수, 0x008000)
        lcd문자(4, 8, 130, "BEST " + 최고각, 0xaa00aa)
        로그(감시모드 + " DIR " + 후보각 + " SCORE " + 점수)
        if (i < 각도목록.length - 1) {
            maqueenPlusV2.pidControlAngle(각도목록[i + 1] - 각도목록[i], maqueenPlusV2.MyInterruption.NotAllowed)
        }
    }
    예약회전각 = 최고각 - 각도목록[각도목록.length - 1]
    마지막판단 = "BEST " + 최고각
    로그(감시모드 + " BEST " + 최고각 + " SCORE " + 최고점수)
    maqueenPlusV2.pidControlAngle(예약회전각, maqueenPlusV2.MyInterruption.NotAllowed)
    회전기록(예약회전각)
    basic.pause(300)
    마지막탐색점수 = 최고점수
    return 최고각
}
```

- 진입 시 현재 헤딩 기준으로 `각도목록[0]`만큼 먼저 회전(기존 `재탐색180()`의
  `pidControlAngle(-90, ...)` 초기 회전과 동일한 역할, 배열 첫 값으로 일반화)
- 루프 안 회전량은 `각도목록[i+1] - 각도목록[i]`로 계산 — 기존처럼 항상 45가
  아니라 호출자가 넘긴 간격(15든 45든)을 그대로 따라간다
- 끝나면 마지막 루프 위치(`각도목록[len-1]`)에서 `최고각`으로 다시 회전해 그
  방향을 보게 만듦(기존 동작과 동일한 `예약회전각` 계산식을 일반화)
- 반환값은 `최고각`(다음 단계로 넘어갈 때 되돌리기 회전에 필요), 점수는
  기존처럼 전역 `마지막탐색점수`에 남겨 호출자가 읽음

## 설계: `재탐색180()` 리팩터링

```typescript
function 재탐색180(탈출모드: boolean): boolean {
    감시모드 = 탈출모드 ? "ESCAPE" : "RESCAN"
    상태 = 탈출모드 ? "탈출탐색" : "재탐색"
    마지막판단 = ""
    maqueenPlusV2.pidControlStop()

    let 최고각 = 각도순회탐색(재탐색각도)
    if (마지막탐색점수 >= 재탐색최소점수 && 방향안전확인()) return true
    maqueenPlusV2.pidControlAngle(-최고각, maqueenPlusV2.MyInterruption.NotAllowed)

    감시모드 = 탈출모드 ? "ESCAPE-FINE" : "RESCAN-FINE"
    최고각 = 각도순회탐색(재탐색각도세밀)
    if (마지막탐색점수 >= 재탐색최소점수 && 방향안전확인()) return true
    maqueenPlusV2.pidControlAngle(-최고각, maqueenPlusV2.MyInterruption.NotAllowed)

    감시모드 = 탈출모드 ? "ESCAPE-BACK-R" : "RESCAN-BACK-R"
    최고각 = 각도순회탐색(재탐색각도세밀후방우)
    if (마지막탐색점수 >= 재탐색최소점수 && 방향안전확인()) return true
    maqueenPlusV2.pidControlAngle(-최고각, maqueenPlusV2.MyInterruption.NotAllowed)

    감시모드 = 탈출모드 ? "ESCAPE-BACK-L" : "RESCAN-BACK-L"
    최고각 = 각도순회탐색(재탐색각도세밀후방좌)
    if (마지막탐색점수 >= 재탐색최소점수 && 방향안전확인()) return true

    로그("ALL 360 FINE STAGES FAILED -> NO SAFE DIR")
    return false
}
```

호출자(`출발준비()` `:936`, `탈출탐색()` `:904`)는 변경하지 않는다. 둘 다 이미
`재탐색180(false)`/`재탐색180(true)`가 `false`를 반환하면 `"NO SAFE DIR"` 처리를
하므로, 4단계를 다 거치고 실패했을 때의 동작은 지금과 동일하게 자연스럽게 이어진다.

## 시간 비용

정상 경로(1단계에서 안전한 방향을 찾는 가장 흔한 경우)는 기존과 동일한 시간이 걸린다.
2~4단계는 1단계가 실패했을 때만(이미 "문제가 생긴" 드문 경로) 추가로 실행되므로, 평소
주행에는 영향이 없다. 최악의 경우(4단계 전부 실행) 회전+측정 5+13+6+6=30회로 늘어나지만,
이는 기존에도 실패 시 `"NO SAFE DIR"`로 바로 멈추던 경로가 "조금 더 찾아보고 멈추는"
경로로 바뀌는 것뿐이라 허용 가능한 트레이드오프로 본다.

## 검증

자동 테스트 러너가 없는 프로젝트라 검증은 기존 패턴을 따른다.

1. 정적: 코드 블록 추출 후 `tsc --noEmit`으로 타입 체크
2. 의사실행 트레이스: 각 단계 배열의 첫/끝 회전량 계산(`목록[0]`, `목록[i+1]-목록[i]`,
   `최고각 - 목록[len-1]`)이 1단계(기존 동작)와 동일한 결과를 내는지 손으로 비교 확인
3. 하드웨어 체크리스트(사람이 실행): 일부러 정면을 거의 다 막은 상태에서 좁은 틈만
   남겨 두고 출발을 시도해, 1단계(45°)로는 못 찾고 2단계(15°) 이상에서 찾아 출발하는지
   확인. 완전히 사방을 막아 4단계 모두 실패할 때 `"NO SAFE DIR"`/X 아이콘이 뜨는지 확인.

## 범위 외

- `초기360탐색()`(최초 8방향 코스 스캔)의 세분화는 범위 밖 — 1단계 코스 스캔은 그대로 둠
- 라이다 각도/높이 보정(`각도진단()`, `ANGLE OK` 등) 로직 자체는 변경하지 않음
- 5x5 LED 레이더 표시(`LED레이더표시()`)는 변경하지 않음
