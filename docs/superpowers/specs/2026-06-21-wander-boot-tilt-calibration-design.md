# 부팅 시 라이더 기울기 보정 흐름 설계 (AUTONOMOUS_WANDER_EXAMPLE.md)

## 배경 / 목적

지금 메인 자율주행 코드는 부팅 후 A 버튼으로 1회성 `바닥보정()`(3·2·1
카운트다운 → Y5/Y6/Y7 raw 표본 + 전체 거리 기준값(`기준값`) 수집 →
`각도진단()`으로 `각도상태`를 "ANGLE OK"/"RAISE SENSOR"/"LOWER SENSOR"/
"LEVEL SENSOR"/"BASE NOISY" 중 하나로 판정)을 1번 실행한 뒤, B 버튼을
눌러야 주행이 시작된다. 단 B는 `보정완료 && 각도상태=="ANGLE OK"`일
때만 실제로 `출발준비()`를 호출하고, 그렇지 않으면 "PRESS A"/"ANGLE NG"
메시지만 띄우고 주행을 막는다.

이 흐름은 "라이더 기울기를 손으로 맞춰가며 실시간으로 보고 싶다"는
요구와 맞지 않는다 — `바닥보정()`은 한 번 찍고 끝나는 스냅샷이라, 기울기
나사를 돌리면서 화면이 계속 갱신되는 걸 보려면 A를 반복해서 눌러야 하고
매번 3·2·1 카운트다운과 완료 아이콘 표시(총 ~4초)를 거쳐야 한다.

이번 설계는 버튼 의미를 다음으로 바꾼다:
- **A** — 지금 상태 그대로 즉시 주행 시작(기울기 OK 여부와 무관하게).
- **B** — 라이더 기울기를 실시간으로 보면서 손으로 조절하는 라이브
  모드로 진입.
- **AB** — 기존과 동일(라이더 높이 100/140/180mm 순환).

## 비목표 (Out of scope)

- 라이더 기울기를 모터/서보로 자동 조절하지 않는다 — 이번에도 사람이
  손으로 조절하고, 코드는 실시간 피드백만 보여준다.
- `각도진단()`의 판정 임계값(Y6 260~520mm, Y7 ≥200mm @ 기본높이 140mm)은
  바꾸지 않는다 — 이미 검증된 값을 그대로 재사용한다.
- 8x8 LCD 레이더맵(`lcd레이더맵표시`)을 5x5 LED 매트릭스로 바꾸는 작업은
  별도 스펙(`2026-06-21-wander-5x5-radar-design.md`, 추후 작성)으로
  분리한다 — 이번 스펙은 부팅/버튼 흐름만 다룬다.

## 현재 구조 참고 (변경 전)

- `바닥보정()` (라인 638-709): 카운트다운 → `전체샘플읽기()` 5회(→
  `기준값` 산출, 장애물 감지 기준 거리) → Y5/Y6/Y7/좌/우 raw 표본 5회
  (→ `각도진단()` 호출) → 여러 주행 상태 변수 리셋 → 완료 아이콘.
- `각도진단(y5목록, y6목록, y7목록, 좌목록, 우목록)` (라인 612-636):
  순수 판정 함수, 입력 리스트만 있으면 길이/노이즈/임계값으로
  `각도상태`/`보정Y5`/`보정Y6`/`보정Y7`/`보정좌하단`/`보정우하단`/
  `보정노이즈`를 갱신. **이 함수는 그대로 재사용한다.**
- `보정요청`/`출발요청`/`높이변경요청` 플래그 + `basic.forever` 메인
  루프(라인 959-995 부근)에서 플래그를 소비.
- `lcd대기표시(강제)` (라인 306-314): 대기 화면에 높이/버튼안내/Y5·Y6·
  Y7/각도상태를 그린다. `보정완료 ? 각도상태 : "PRESS A CAL"` 삼항식
  포함.
- `로봇초기화()` (라인 316-330): 부팅 화면에 "A = CALIBRATE" / "B =
  START" 안내 표시.

## 새 동작 흐름

1. **A 버튼** (주행 중이 아닐 때): `출발요청 = true`만 설정(기존과
   동일한 패턴 — 버튼 핸들러에서 무거운 작업을 직접 하지 않고 메인
   루프가 처리). 라이브 모드 진입 중이었다면 같이 `기울기보정모드중 =
   false`로 끈다.
2. **B 버튼** (주행 중이 아닐 때): `기울기보정모드중 = true`로 설정.
3. **메인 루프**의 `!주행시작됨` 분기를 다음 순서로 재구성한다(먼저
   `출발요청`을 확인해 A가 항상 우선하도록):
   - `출발요청`이 true면: 플래그를 끄고 `기울기보정모드중`도 끈 뒤,
     **변경 없이** `바닥보정()` → `출발준비()`를 그대로 호출한다(이제
     `각도상태` 값과 무관하게 항상 호출 — "OK여야 시작" 게이트 제거).
   - 아니고 `기울기보정모드중`이 true면: `기울기보정라이브틱()`을 호출한
     뒤 `기울기보정틱대기ms`만큼 쉬고 그 루프 반복을 끝낸다(`return`).
   - 둘 다 아니면: 기존처럼 `lcd대기표시(false)`만 호출한다.
4. **`기울기보정라이브틱()`** (신규 함수): `바닥보정()`의 Y5/Y6/Y7/좌/우
   raw 표본 수집 루프(라인 659-684, 5회 반복, 매회 10개 지점 읍기)를
   그대로 복사해 가져오되, `전체샘플읽기()`/`기준값` 계산, 3·2·1
   카운트다운, 상태 변수 리셋, 완료 아이콘 표시는 전부 뺀다. 5회 반복을
   유지하는 이유는 `각도진단()` 내부에 `y6목록.length < 3` 같은
   "표본이 충분한가" 가드가 있어서, 표본 수가 너무 적으면(예: 1회만
   읍으면 리스트 길이가 2 이하) 항상 "LOWER SENSOR"로 잘못 판정되기
   때문이다 — 5회 반복은 그 가드를 만족시키기 위한 최소 조건이고,
   `각도진단()` 자체는 수정하지 않는다.
   표본 수집 후 `각도진단(y5목록, y6목록, y7목록, 좌목록, 우목록)`을
   호출해 `각도상태`/`보정Y5`/`보정Y6`/`보정Y7`을 갱신하고, LCD에
   다음 5줄을 다시 그린다(매 호출마다 강제 갱신, 기존 `lcd문자` 헬퍼
   재사용):
   1. `"TILT CAL LIVE"`
   2. `"ADJUST TILT BY HAND"`
   3. `"PRESS A TO START"`
   4. `"Y5 " + 보정Y5 + " Y6 " + 보정Y6 + " Y7 " + 보정Y7"` (기존
      `lcd대기표시`의 4번째 줄과 동일한 포맷)
   5. `각도상태`, 색은 기존 `각도색()` 그대로 재사용
5. **AB(높이 변경)**: 라이브 모드 중에도 그대로 허용한다. 높이가
   바뀌면 `높이비례값()`이 새 `라이더높이mm` 기준으로 다시 계산되므로,
   다음 `기울기보정라이브틱()` 호출부터 Y6/Y7 판정 임계값이 새 높이
   기준으로 자동 반영된다(따로 손볼 코드 없음). `높이변경()`은 현재도
   `보정요청`/`출발요청`을 끄는데, 여기에 `기울기보정모드중 = false`는
   추가하지 않는다 — 높이를 바꾼 뒤에도 라이브 화면을 계속 보면서
   조절하는 게 자연스럽기 때문이다(다음 루프 틱에서 라이브 화면이 새
   높이로 갱신되어 다시 보인다).

## 화면 안내 문구 변경

- `로봇초기화()` (라인 327-329): `"A = CALIBRATE"` → `"A = START"`,
  `"B = START"` → `"B = TILT CAL"`.
- `lcd대기표시()` (라인 309-313): 2번째 줄 `"A CAL  B START"` →
  `"A START  B TILT CAL"`. 5번째 줄의 `보정완료 ? 각도상태 : "PRESS A
  CAL"` 삼항식은 단순히 `각도상태`로 바꾼다(초기값이 이미 `"ANGLE ?"`
  이므로 "아직 보정 안 함" 의미가 자연스럽게 유지됨).

## 정리(Cleanup)할 죽은 코드

이 변경으로 `보정완료` 변수가 더 이상 어디서도 읍히지 않게 된다(쓰기만
남음: `높이변경()`의 리셋, `바닥보정()`의 `true` 설정). 변수 선언과 그
두 대입문을 삭제한다. 메인 루프의 다음 블록(현재 출발 게이트 + 실패
메시지)도 통째로 삭제한다(라인 982-990 부근):

```typescript
if (출발요청 && 보정완료 && 각도상태 == "ANGLE OK") {
    로그("START PRESSED -> 출발준비")
    출발준비()
} else {
    let 이전판단 = 마지막판단
    if (출발요청 && !보정완료) 마지막판단 = "PRESS A"
    if (출발요청 && 보정완료 && 각도상태 != "ANGLE OK") 마지막판단 = "ANGLE NG"
    if (출발요청 && 마지막판단 != 이전판단) 로그("CANNOT START: " + 마지막판단)
    출발요청 = false
    lcd대기표시(false)
}
```

위 블록은 "새 동작 흐름" 3번에서 설명한 3-way 분기(출발요청 / 기울기
보정모드중 / 대기)로 대체된다.

## 새 전역 상태/상수

```
let 기울기보정모드중 = false
const 기울기보정틱대기ms = 100
```

## 에러 처리

- `각도진단()`이 "BASE NOISY"로 판정해도 라이브 모드는 멈추지 않는다
  — 사람이 화면에서 "BASE NOISY"를 보고 무엇을 고쳐야 할지 판단하면
  된다(기존에도 주행 중 BASE NOISY면 정지하는 별도 로직(라인 1028
  부근)이 있지만, 그건 주행 중 상태이고 이번 라이브 모드는 주행 전
  단계라 영향 없음).
- A를 눌러 시작할 때 `바닥보정()`이 "BASE NOISY"로 판정하더라도, 이번
  설계는 게이트를 없앴으므로 그대로 `출발준비()`가 호출된다. 이건
  의도된 동작이다 — 사람이 라이브 화면으로 이미 충분히 보정했다고
  판단하고 A를 눌렀다는 전제를 신뢰한다.

## 테스트 / 검증 방법

이 프로젝트에는 자동화 테스트 러너가 없으므로(MakeCode/PXT 하드웨어
펌웨어), 검증은 (1) tsc 정적 타입체크와 (2) 실제 하드웨어 체크리스트로
구성된다.

- 부팅 직후 LCD에 "A = START" / "B = TILT CAL" 안내가 보이는가.
- B를 누르면 LCD가 "TILT CAL LIVE" 화면으로 바뀌고 Y5/Y6/Y7 + 각도상태가
  주기적으로(약 0.5~0.6초 간격) 갱신되는가.
- 라이브 모드 중 라이더 기울기를 손으로 바꾸면 Y6/Y7 값과 각도상태
  문구가 그에 맞게 바뀌는가.
- 라이브 모드 중 AB를 누르면 높이가 바뀌고, 이후 Y6/Y7 판정 기준이
  새 높이로 갱신되는가(예: 높이를 낮추면 같은 raw 값이라도 "RAISE
  SENSOR"였던 게 "ANGLE OK"로 바뀔 수 있음 — `높이비례값` 스케일링
  때문).
- 라이브 모드 중이든 대기 화면이든 A를 누르면 항상 3·2·1 카운트다운 →
  보정 완료 아이콘 → 주행 시작까지 이어지는가(각도상태와 무관하게).
- A로 시작한 직후 `기준값`(장애물 감지 기준)이 정상적으로 설정되어
  기존 회피 로직이 그대로 동작하는가(기존 동작과 동일해야 함 —
  `바닥보정()` 자체는 안 바꿨으므로).
