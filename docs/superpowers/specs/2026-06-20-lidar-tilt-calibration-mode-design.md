# 라이더 기울기(tilt) 실시간 보정 모드 설계

## 배경 / 목적

`PANEL_CALIBRATION_TEST.md`는 라이더 높이가 물리적으로 140mm
(`기본라이더높이mm`)에 고정되어 있다고 가정하지만, 실제로 그 높이에서
**기울기(아래로 얼마나 숙였는지)가 올바른지**는 사람이 손으로 조절해야
한다. 지금까지는 이 조절을 도와줄 실시간 피드백이 없었다 — 각도체크
모드(`각도체크시작`)는 한 자리에서 5회 측정해 raw 값을 한 번 라디오로
보내고 끝나는 1회성 배치 동작이라, 손으로 기울기를 조금씩 바꾸면서
실시간으로 값이 어떻게 변하는지 보는 용도로는 맞지 않는다.

목표는 로봇을 박스 패널 앞에 두고 트리거를 누르면, 로봇이 **자동으로
10cm 후진**한 뒤 라이더 **최하단 행(Y7)**의 8개 컬럼 값을 **연속으로
LCD에 갱신**해서, 사람이 손으로 라이더 기울기를 조절하면서 그 자리에서
바로 결과를 보고 맞출 수 있게 하는 것이다. 로봇 키(약 140mm)보다 높은
지점의 데이터는 이 보정에 필요 없으므로 보지 않는다.

이 모드는 `PANEL_CALIBRATION_TEST.md`에 새로 추가되는 네 번째 진입점이며,
기존 세 모드(각도체크/패널테스트/회전테스트)와 독립적으로 동작한다.

## 비목표 (Out of scope)

- 라이더 기울기를 모터/서보로 자동 조절하지 않는다 — 펌웨어
  (`maqueenPlusV3.ts`)에 그런 액추에이터 제어 함수가 없고, 조절은 항상
  사람이 손으로 한다. 이 모드는 손 조절을 돕는 **실시간 게이지**일 뿐이다.
- 라디오 전송을 하지 않는다 — 이 모드는 그 자리에서 즉시 보는 로컬
  피드백이 목적이라 무선 왕복 지연이 필요 없다.
- 자동 OK/NG 판정을 하지 않는다 — 목표거리/컬럼편차 계산값이 둘 다
  근사값(센서 장착 높이와 후진거리를 기반으로 한 추정)이라, 기준을
  하드코딩해서 "맞다/틀리다"로 단정하면 오판할 수 있다. 계산된 숫자만
  보여주고 최종 판단은 사람이 한다.

## 동작 흐름

1. **트리거**: 로고(logo) 터치센서
   (`input.onLogoEvent(TouchButtonEvent.Pressed, ...)`). 다른 모드가
   진행 중이면(`시작됨 == true`) 무시한다.
2. `시작됨 = true`로 설정하고 `maqueenPlusV2.pidControlDistance(
   maqueenPlusV2.SpeedDirection.SpeedCCW, 보정후진거리cm,
   maqueenPlusV2.MyInterruption.NotAllowed)`로 10cm 후진한다(기존
   패널테스트가 1cm씩 후진할 때 쓰는 것과 같은 방향 상수).
3. 후진이 끝나면 라이브 루프 시작. A 버튼이 눌릴 때까지 다음을
   반복한다(틱마다 `기울기갱신지연ms` 만큼 `basic.pause`):
   - Y7 행(8x8 그리드의 마지막 행, 가장 아래를 보는 행)의 8개 컬럼을
     `지점읽기(x, 7)`로 **raw 단일 읍기**한다(노이즈 제거용 반복 읍기는
     쓰지 않는다 — 응답성이 정확도보다 우선).
   - 유효값(0이 아닌 값) 목록의 **median**과, 그 목록의 **max−min
     편차**를 계산한다(유효값이 2개 미만이면 편차는 0으로 둔다).
   - **목표거리 대비 차이**를 `median − 목표거리mm`로 계산한다(부호를
     남겨서 방향 힌트로 사용 — 양수면 더 멀리/위를 보고 있다는 뜻).
   - LCD에 5줄로 갱신 표시(아래 LCD 표시 섹션 참고).
4. A 버튼을 누르면 즉시 루프를 종료하고 `시작됨 = false`로 되돌린 뒤
   대기 화면(`lcd대기표시()`)으로 복귀한다. 기존 A 버튼 핸들러
   (`if (시작됨) { 중단요청 = true; maqueenPlusV2.pidControlStop() }`)를
   그대로 재사용하며, 루프는 매 틱 `중단요청` 플래그를 확인해 빠진다.
   후진 동작 외에는 로봇이 움직이지 않으므로 `pidControlStop()` 호출은
   사실상 동작 없음(no-op)이다.

## 목표거리 계산

라이더가 장착 높이 `기본라이더높이mm`(140mm)에서 수평으로
`보정후진거리mm`(100mm) 떨어진 지점의 **바닥 높이(0mm)** 를 보고 있다면,
센서에서 그 지점까지의 빗변 거리는 피타고라스 정리로 다음과 같다:

```
목표거리mm = Math.round(Math.sqrt(보정후진거리mm * 보정후진거리mm + 기본라이더높이mm * 기본라이더높이mm))
           ≈ Math.round(Math.sqrt(100*100 + 140*140))
           ≈ 172
```

이 값은 상수로 한 번 계산해 두고 코드에서 재사용한다. `기본라이더높이mm`가
바뀌면(다른 높이로 재실험) 이 상수도 같이 다시 계산되도록 리터럴이 아닌
수식으로 작성한다.

## LCD 표시

매 틱 다음 5줄을 갱신한다(판정 문구/색 구분 없음, 숫자만):

1. `TILT CAL` (고정 타이틀)
2. 8개 raw 값을 공백으로 구분한 한 줄, 예: `118 120 119 121 117 119 120 118`
3. `DIFF {부호}{차이}mm` — `median − 목표거리mm`, 예: `DIFF +12mm`
4. `SPREAD {편차}mm` — 유효값 max−min, 예: `SPREAD 4mm`
5. `A=STOP` (고정 안내)

## 새 전역 상수

```
보정후진거리cm = 10            // pidControlDistance 단위(cm)
보정후진거리mm = 100            // 목표거리 계산용(mm)
거리허용오차mm = 15             // 참고용 — LCD에는 안 쓰지만 사람이 보정 시 감안할 여유값으로 문서에 명시
컬럼편차참고mm = 10             // 참고용 — 위와 동일 목적
목표거리mm = Math.round(Math.sqrt(보정후진거리mm*보정후진거리mm + 기본라이더높이mm*기본라이더높이mm))
기울기갱신지연ms = 150
```

`거리허용오차mm`/`컬럼편차참고mm`는 LCD 표시나 판정 로직에는 쓰이지 않고,
문서의 "실행 전 준비물"/"검증 체크리스트"에서 사람이 "대략 이 범위
안이면 괜찮다고 보면 된다"는 참고 기준으로만 언급한다(자동 판정 금지
원칙과 모순되지 않도록, 코드가 아니라 문서 텍스트에만 등장).

## 새 함수

- `기울기보정시작(): void` — 로고 터치 핸들러가 호출. 시작됨 체크,
  10cm 후진, 라이브 루프(아래 `기울기보정틱`을 반복 호출), 종료 시 정리.
- `기울기보정틱(): void` — 한 번의 측정+LCD 갱신을 담당(Y7 8개 컬럼
  raw 읍기, median/편차/차이 계산, LCD 5줄 갱신).
- `lcd기울기보정표시(값들: number[], 차이mm: number, 편차mm: number): void`
  — 기존 `lcd문자` 헬퍼를 그대로 사용해 5줄을 그린다.

기존 `정렬삽입`/`중앙값`/`지점읽기`/`lcd문자` 등 헬퍼를 그대로 재사용하고
새로 추가하는 함수는 위 3개뿐이다.

## 에러 처리

- Y7 8개 컬럼 중 유효값(0 아님)이 하나도 없으면 median을 0, 편차를
  0, 차이를 `0 − 목표거리mm`로 그대로 계산해 보여준다(별도 N/A 처리
  없음 — 0이 나온다는 사실 자체가 "센서가 아무것도 못 본다"는 유효한
  피드백이기 때문).
- 로고 터치가 다른 모드 진행 중에 들어오면 무시한다(`시작됨` 플래그로
  방지, 기존 B/AB 핸들러와 동일한 가드 패턴).

## 문서화 (PANEL_CALIBRATION_TEST.md 갱신 범위)

- "사용 방법"에 로고 터치 단계 추가.
- "로그 형식 설명" 섹션은 라디오를 안 쓰므로 그대로 두고, 대신 LCD
  표시 형식을 설명하는 절을 새로 추가.
- 이 모드가 다른 모드와 달리 "차이/편차 숫자를 계산해서 보여준다"는
  점과, 그 이유(로컬 실시간 보정 보조 목적이며 자동 판정이 아님)를
  "이 스크립트는 계산을 하지 않는다" 섹션 옆에 예외로 명시한다.
- "실행 전 준비물"에 로고 터치 트리거 안내와, `거리허용오차mm`/
  `컬럼편차참고mm`를 참고 기준으로 언급.
- "실행 후 하드웨어 검증 체크리스트"에 로고 터치 → 10cm 후진 → 실시간
  LCD 갱신 → A로 종료까지 항목 추가.

## 테스트 / 검증 방법

이 프로젝트에는 자동화된 하드웨어 테스트 러너가 없으므로(MakeCode/PXT
펌웨어), 검증은 (1) 코드 정적 점검(tsc 타입체크, 기존 패턴과의 일치
여부)과 (2) 사람이 실제 로봇으로 실행하는 체크리스트로 구성된다.

- 로고를 터치하면 로봇이 정확히 10cm 후진하는지 줄자로 대조.
- 후진 후 LCD가 150ms 주기로 갱신되며 8개 raw 값이 출력되는지 확인.
- 라이더 기울기를 손으로 바꿔가며 `DIFF`/`SPREAD` 값이 그에 따라
  눈에 보이게 변하는지 확인(목표거리에 가까워질수록 `DIFF`가 0에
  가까워지고, 평평하게 모서리를 볼수록 `SPREAD`가 작아짐).
- A 버튼을 누르면 즉시 루프가 멈추고 대기 화면으로 돌아가는지 확인.
- 다른 모드(B/AB) 진행 중에 로고를 터치해도 아무 일도 일어나지 않는지
  확인.
